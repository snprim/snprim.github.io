title: Accessing NHL Data via APIs and Creating Functions
---

In this project, I created functions to demonstrate how to retrieve information via APIs using two APIs of the National Hockey League, clean up or combine data, and then doing exploratory data analysis. All the steps are presented in [my vignette](https://snprim.github.io/ST558_project_1/). This project has shown me the power and challenges of using APIs. 

APIs provide a convenient method for information sharing, and the steps to retrieve information is rather straightforward. However, I have come to appreciate good documentations. Each API is a different system. Each API requires the users to understand terms about the subject to know which endpoints are relevant. Moreover, the data structure provided in each endpoint could be very different, and knowing what data to return is the key to good API functions. After the data is cleaned up, another challenge is to know what aspects of the data to explore. Without having specific questions about NHL statistics, exploratory data analysis can be tough. 

When I approach another project to access APIs in the future, I will spend more time reading documentations and understanding data structures. While writing the right code to retrieve data isn’t difficult, understanding what the data means, knowing what data to retrieve, and assessing whether this is what the user might want are crucial aspects that require more domain knowledge. Knowing what data is relevant also helps me decide what data to remove; analysis is much easier when only relevant variables are kept. 

The difficult part of the logic and programming is perhaps writing functions for repeated usage and potentially other users. Retrieving a specific endpoint, or a subset of an endpoint, for myself is manageable; as long as the information retrieved is what I need, the task is done. Writing functions for APIs is a completely different story. The functions need to consider as many possible scenarios as possible to avoid errors; furthermore, keeping track of all the endpoints and deciding what arguments the users (or the future me) should provide are no easy tasks. Adding an index to connect team names and Ids is technically easy but requires some experience in creating functions with the users in mind. This experience helps me refine my programming skills and think about the overall planning.

The project has several big takeaways. First of all, writing code isn’t hard, but writing several hundred lines of code that all work together is hard. Secondly, writing functions for repeated usage is much more difficult than writing code to retrieve data once. Thirdly, using data sources requires understanding of the specific source and domain knowledge. Lastly, exploratory data analysis is open-ended and requires background knowledge; when done correctly, summaries and plots can tell interesting and meaningful stories. Although this is a time-consuming project, the process is quite fun, and the lessons learned are worth the time and efforts!
